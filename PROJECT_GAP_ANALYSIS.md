## 结构与架构缺陷

### 1.1 缺失的客户端应用 ❌

**问题**：
- README 中声称支持"Web 前端"与"Electron 桌面端"
- 但 `apps/` 目录实际不存在
- AGENTS.md 中提到 `apps/dsa-web/` 与 `apps/dsa-desktop/` 目录，但这些目录为空或不存在

**影响**：
- 用户期望获得不到承诺的 UI 交互
- Web 工作台与桌面应用成为虚拟功能
- README 中发布亮点提及"Web 工作台完成一轮 UI 统一与双主题升级"但无对应代码

**验证**：
```bash
ls -la apps/  # 目录不存在或为空
```

**建议**：
- 确认这些应用是否在其他分支或仓库
- 如在主分支缺失，应明确说明当前只支持命令行/API 模式
- 或在项目文档中标记"前端应用开发中"（WIP）

---

### 1.2 代码臃肿与职责划分不清 📦

**核心问题**：
- **[src/notification.py](src/notification.py)** (~83KB)
  - Markdown 拼接逻辑直接内聚在 NotificationService 中
  - 表格生成、Emoji 映射、格式化分散在各处
  - 支持 10+ 种通知渠道但代码复杂度线性增长

- **[data_provider/base.py](data_provider/base.py)** (~96KB)
  - 数据获取调度逻辑、缓存管理、故障切换混合
  - DataFetcherManager、实时行情缓存、Fallback 机制耦合

**影响**：
- 代码难以测试（单元测试行数 ↑）
- 新增渠道时改动面大（紧耦合）
- 错误追踪困难（职责不清）
- 代码复用难（函数粒度过粗）

**已识别方案**（见 [docs/NEXT_OPTIMIZATIONS.md](docs/NEXT_OPTIMIZATIONS.md)）：
1. 将 `generate_daily_report` 等渲染方法迁至 `src/notification/renderer.py`
2. 格式化工具迁至 `src/notification/utils.py`
3. 创建 `data_provider/manager.py`、`data_provider/cache.py`、`data_provider/models.py`

**优先级**：🟠 中高（影响可维护性）

---

### 1.3 异步化不完整 ⚡

**问题**：
- 搜索 Provider 中仍有同步实现：
  - `Bocha`、`Brave`、`Exa`、`MiniMax` 使用同步 `requests` 库
  - 未统一迁至异步 `httpx` 客户端

- 核心 Pipeline 中仍存在 `asyncio.to_thread` 包装：
  - 技术面分析（TA 计算）
  - 筹码分布计算
  - 可能的 CPU 密集运算

**影响**：
- 多搜索引擎自动切换时产生 I/O 阻塞
- 高并发市场行情查询时吞吐量下降
- 事件循环被线程池调用阻断

**建议**：
1. 使用 `httpx` 异步库重写各 Provider 的 `_do_search_async()`
2. 评估 CPU 密集操作是否应迁至 `ProcessPoolExecutor`
3. 补充异步化后的性能基准测试

**优先级**：🔴 高（影响性能）

## 质量与可靠性缺陷

### 2.1 测试覆盖不完整 🧪

**现状**：
- 60+ 个测试文件（`tests/` 目录）
- 但关键模块缺少测试：
  - ❌ `src/core/pipeline.py` —— 主分析流程
  - ❌ `src/services/stock_analyzer.py` —— 股票分析核心
  - ❌ `api/` —— FastAPI 应用端到端测试
  - ❌ 数据源 Fallback 链路集成测试
  - ❌ 并发执行相关的压力测试

**测试运行现状**：
```bash
pytest -m "not network"  # 跳过网络测试
```
- 网络测试被 CI 跳过（`network-smoke.yml` 仅作观测）
- 真实数据源验证缺失

**测试标记**：
- `@pytest.mark.network` 存在
- 但运行策略不明确（本地开发 / CI 分流）

**建议**：
1. 补充 `src/core/pipeline.py` 的单元测试（mocking 外部依赖）
2. 添加集成测试覆盖主流程的各分支（dry-run 模式）
3. 建立测试覆盖率基准（目标 >80%）
4. 在 CI 中分离"离线测试"与"网络测试"流水线

**优先级**：🔴 高

---

### 2.2 错误恢复机制不足 🛡️

**问题**：
- 缺少**统一的异步重试策略**
  - 网络超时、速率限制、服务不可用 的处理分散
  - 单一数据源失败可能拖垮整个分析链路
  - 缺少退避算法（exponential backoff）

- 现存重试使用简陋：
  - `tenacity` 库已在 `requirements.txt` 但使用不充分
  - 异步装饰器未被广泛应用

**CHANGELOG 中已修复但仍需注意**：
- v3.12.0 修复了"并发执行时共享状态缺少统一加锁"
- 但仍可能有其他竞态条件隐藏

**建议**：
1. 引入 `AsyncRetryPolicy` 统一策略：
   - 网络超时 → 指数退避 3 次
   - 速率限制 → 延迟 + 指数退避
   - 模型崩溃 → 切换备选模型
2. 所有 I/O 操作应使用统一的 `@async_retry` 装饰器
3. 补充错误恢复的集成测试

**优先级**：🟠 中高

---

### 2.3 并发安全问题 🔒

**已修复问题**（v3.12.0）：
- ✅ "并发执行时共享状态补齐统一加锁"
- ✅ "单股推送模式不再并发复用共享通知实例"

**可能仍存在的隐患**：
- SQLite 锁竞争（虽然 v3.12.0 增强了 WAL + busy_timeout）
- 多线程日志输出是否加锁
- 缓存字典在并发读写时的安全性

**建议**：
1. 补充并发场景的压力测试
2. 使用 `threading` 工具进行死锁检测
3. 在文档中明确并发模型（asyncio/threading/multiprocessing）

**优先级**：🟠 中

## 性能优化不足

### 3.1 缓存策略不完善 💾

**问题**：
- 实时行情缓存逻辑与 `data_provider/` 深度耦合
  - 难以独立验证缓存命中率
  - 缺少 TTL 策略文档

- 缺少 HTTP 缓存头策略
  - Web API 响应未利用 `ETag` / `Last-Modified`
  - 前端无法有效缓存

- 缺少预热机制
  - 应用启动时不预加载常用数据
  - 用户首次查询经历完整延迟

**已修复**（v3.12.0）：
- ✅ "实时行情降级提示收口为单次告警"

**建议**：
1. 提取 `data_provider/cache.py` 为独立模块
2. 实现 TTL 配置与简单的 LRU cache
3. 在 Web 层添加 `Cache-Control` 头
4. 应用启动时预热热门股票数据

**优先级**：🟡 低-中

---

### 3.2 搜索性能尾延迟 🔍

**问题**：
- 新闻搜索返回后逐条同步抓取正文
  - 阻塞 Event Loop
  - 单慢链接拖累整体

**改进跟踪**（v3.12.0）：
- ✅ "SerpAPI 正文补抓范围收敛"（仅高位低质补抓）
- ✅ "A 股中文资讯搜索恢复中文优先"

**仍可优化**：
- 正文补抓应异步化
- 应设置最大并发抓取数限制
- 单条新闻超时应快速失败

**优先级**：🟡 中

## 数据源与适配器缺陷

### 7.1 数据源优先级与 Fallback 不清晰 📊

**优先级链路**（`requirements.txt` 标注）：
```
Priority 0: efinance（东方财富，最高优先）
Priority 1: akshare
Priority 2: tushare / pytdx
Priority 3: baostock
Priority 4: yfinance（Yahoo Finance, 最后 fallback）
```

**问题**：
- 优先级在代码中不清晰（可能需翻代码）
- 单一数据源失败的切换逻辑分散
- 长桥（Longbridge）集成刚完成，稳定性未充分验证

**长桥集成**（v3.12.0 新增）：
- ✅ 美股/港股 日线与实时行情已优先使用
- ⚠️ 但缺少详细的 integration test
- ⚠️ 配置切换与降级路径不明确

**建议**：
1. 明确记录每条 Fallback 链路（文档 + 代码注释）
2. 补充数据源 fallback 的集成测试
3. 实现"数据源健康检查"机制
4. 在 Web 设置页显示当前活跃的数据源

**优先级**：🟠 中

### 7.2 数据一致性问题 🔄

**设计原则**：
- 美股历史数据与实时行情统一用 YFinance（保证复权一致）

**问题**：
- 不同数据源返回格式不统一
  - 字段名、时间戳格式、价格精度 可能不同
  - 缺少统一的 data normalization 层

- 汇率刷新机制不稳定
  - v3.12.0 改进了"断点续传按市场时区判断交易日"
  - 但汇率缓存与更新逻辑仍未明确

**建议**：
1. 创建 `data_provider/normalizer.py` 统一数据格式
2. 定义清晰的数据质量检验函数
3. 为汇率、股票名称等关键数据实现缓存策略
4. 补充"数据一致性检验"测试用例

**优先级**：🟠 中

2. 在 API 层实现全局速率限制
3. 为 Web UI 补充 CSRF token 验证
4. 定期进行安全审计（OWASP Top 10）

**优先级**：🟠 中

### 8.2 数据保护 🔒

**问题**：
- SQLite 本地存储未加密
  - `data/stock_analysis.db` 存储敏感的分析历史
  - 物理访问可直接读取

- API Key 在日志中可能泄露
  - 虽然有脱敏逻辑但不清晰

- `.env` 文件管理缺少加密建议
  - 用户可能将`.env`提交到版本控制

**建议**：
1. 为敏感配置提供加密选项（如 `cryptography` 库）
2. 补充".env 安全管理"最佳实践文档
3. 在 GitHub 配置中启用 `.env` 检测（pre-commit hook）
4. 为 SQLite 考虑启用 SQLCipher（可选）

**优先级**：🟡 中

### 9.2 配置交互复杂 ⚙️
